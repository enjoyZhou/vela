# 定稿后重新编辑设计

## 背景

当前系统将 `finalized` 视为章节创作链路中的稳定里程碑：

- 编辑器在 `finalized`/`archived` 状态下强制只读。
- “正文章节”列表只展示 `status === 'finalized'` 的草稿。
- 后续章节写稿要求前一章已经定稿。
- 定稿会触发正文文件写出、知识库入库、章节要点、角色状态更新等后处理。

现状的问题是：

- 用户在正常界面下无法修正已经定稿的最新章节。
- 如果未来直接放开 `finalized` 的可编辑能力，会让正文内容和定稿后处理结果脱钩。
- 历史章节回溯修改会破坏后续章节依赖的线性状态链。

因此，本设计的目标不是“让定稿态永久可写”，而是提供一条受控的返工路径。

## 目标

- 仅允许最新定稿章节进入重新编辑。
- 重新编辑后，章节退出稳定定稿态，进入“待重新定稿”状态。
- 用户完成修改后，显式触发“重新定稿”以重建该章的定稿产物。
- 禁止历史定稿章节返工，避免破坏后续章节上下文和角色状态链。

## 非目标

- 不支持历史定稿章节回溯返工。
- 不在“确认重新编辑”后自动重跑定稿流程。
- 不在本阶段引入“从某一章开始重建全部后续章节”的能力。
- 不修改现有定稿后处理主流程的核心编排。

## 核心决策

### 决策 1：仅最新定稿章节允许重新编辑

原因：

- 当前 `guardRepairPostProcess()` 已经明确限制“只允许修复最新定稿章节”，说明系统默认后处理和角色状态是线性演化的。
- 允许历史章节返工会直接挑战“后续章节基于前章定稿结果生成”的基础假设。

落地方式：

- 新增 `guardReopenFinalizedChapter(chapterNumber)`。
- 内部复用“最大定稿章节号”校验逻辑。
- 若不是最新定稿章节，直接阻止重新编辑。

### 决策 2：重新编辑后退出 `finalized`

原因：

- `finalized` 必须继续代表“稳定可消费的章节基线”。
- 如果允许 `finalized` 长期可写，会让 UI、正文投影、后处理状态和用户心智全部变得模糊。

落地方式：

- 点击“重新编辑”后，将草稿状态从 `finalized` 改为 `revised`。
- UI 不直接展示“已修稿”，而是以“待重新定稿”作为文案表达。
- 重新定稿前，该章节不再被视为稳定终稿。

### 决策 3：重新定稿继续复用现有定稿流程

原因：

- 现有定稿流程已经负责正文同步、物理文件写出、后处理重跑和统一刷新，不需要重写。
- 返工后的核心诉求是重新建立该章终稿，而不是构造一条全新的工作流。

落地方式：

- 返工后的“重新定稿”继续调用现有 `createFinalizeWorkflow()`。
- 在 UI 文案、确认文案和日志文案中明确这是一次“重新定稿”。

## 行为规则

### 规则矩阵

| 场景 | 是否允许重新编辑 | 是否允许保存 | 是否允许重新定稿 | 说明 |
| --- | --- | --- | --- | --- |
| 最新定稿章节 | 是 | 是 | 是 | 返工后状态退回 `revised`，需显式重新定稿 |
| 非最新定稿章节 | 否 | 否 | 否 | 防止破坏后续章节依赖链 |
| 未定稿章节 | 不涉及 | 是 | 是 | 维持现有逻辑 |
| 重新编辑中的最新章节 | 不再显示“重新编辑” | 是 | 是 | UI 显示为“待重新定稿” |

### 状态流转

正常链路：

`draft -> revised/reviewed -> finalized`

新增返工链路：

`finalized -> revised(待重新定稿) -> finalized`

说明：

- 本期不新增数据库状态值，复用 `revised` 作为返工后的中间态。
- 为避免语义混乱，UI 需要按场景显示“待重新定稿”。

## 交互设计

### 已定稿章节

在 `DraftEditor` 的已定稿只读区域增加“重新编辑”按钮。

交互流程：

1. 用户点击“重新编辑”。
2. 先执行 `guardReopenFinalizedChapter(meta.chapterNumber)`。
3. 通过后弹出确认框。
4. 用户确认后将状态切回 `revised`。
5. 编辑器恢复可写。
6. 顶部按钮区恢复“保存 / AI 修稿 / AI 审稿 / 重新定稿”。

确认文案建议：

- 标题：`确认重新编辑`
- 正文：
  - `重新编辑后，本章将退出稳定定稿状态。`
  - `已有的定稿后处理结果将视为过期。`
  - `完成修改后，需要重新定稿才能继续作为后续章节的稳定上下文。`

### 返工中的章节

返工中的章节在编辑区表现为普通可编辑草稿，但状态文案改为“待重新定稿”。

按钮策略：

- 保留保存按钮。
- 保留 AI 修稿和 AI 审稿。
- 将“定稿”按钮文案改成“重新定稿”。

### 非最新定稿章节

有两种实现方式：

- 隐藏“重新编辑”按钮。
- 显示禁用按钮并给出原因提示。

推荐第二种，因为能更好地传达“为什么不能改”。

提示文案建议：

- `仅最新定稿章节允许重新编辑。`
- `回溯修改历史定稿章节会破坏后续章节的角色状态和上下文链。`

## 实现设计

### 1. `src/services/workflow-guards.ts`

新增函数：

- `guardReopenFinalizedChapter(chapterNumber: number): Promise<GuardResult>`

职责：

- 确认项目已打开。
- 查询最大定稿章节号。
- 若没有已定稿章节，返回失败。
- 若当前章节不是最大定稿章节，返回失败并说明原因。
- 否则返回成功。

实现建议：

- 复用 `db:draft-get-max-finalized-chapter`。
- 错误文案风格与 `guardRepairPostProcess()` 保持一致。

### 2. `src/components/editor/DraftEditor.tsx`

新增函数：

- `doReopenFinalize()`

职责：

- 仅在 `meta.status === 'finalized'` 时可触发。
- 调用 `guardReopenFinalizedChapter(meta.chapterNumber)`。
- 弹出确认框。
- 确认后调用状态更新接口，将当前草稿状态更新为 `revised`。
- 更新本地 `meta`，使当前编辑器立即从只读切为可编辑。
- 视需要刷新草稿列表和相关面板。

现有逻辑需要调整：

- `isReadonly` 维持 `finalized || archived` 判断，不放开 `finalized` 的直接可编辑。
- 已定稿只读区增加“重新编辑”按钮。
- 顶部按钮区里，“定稿”按钮根据上下文展示为“定稿”或“重新定稿”。
- 定稿确认文案从“定稿后不再支持修改”改为更准确的表述。

建议增加派生变量：

- `const isReopenCandidate = status === 'finalized'`
- `const isPendingReFinalize = status === 'revised' && meta?.wasFinalizedBefore === true`

注意：

- 如果本期不引入 `wasFinalizedBefore` 这类额外标记，则“重新定稿”按钮文案可以简化为：当章节存在定稿后处理遗留状态或最近一次状态由 `finalized` 退回时显示“重新定稿”。若难以可靠判断，可接受本期统一显示“定稿”，仅在确认弹窗中说明“将重新建立终稿产物”。

### 3. `src/shared/draft-status.ts`

需要调整：

- 保持 `DraftStatus` 不变，继续复用 `revised`。
- 继续保留内部基础文案 `已修稿`，避免波及过大。

UI 显示策略：

- 具体是否显示“待重新定稿”，放到消费侧组件判断，不在共享常量层写死。
- 这样可以避免普通修稿和“终稿返工”共享一个状态值却必须共用一个标签的问题。

### 4. `src/components/panels/sidebar/DraftBoxGroup.tsx`

需要调整：

- 对 `draft.status === 'revised'` 的章节，结合额外上下文判断是否显示为“待重新定稿”。
- 若本期没有额外字段可判断返工来源，则本组件可以暂时维持“已修稿”，优先保证功能上线。
- 已定稿图标仍然只在 `status === 'finalized'` 时展示。

建议：

- 如果后续准备做第二阶段增强，可为 `drafts` 表或内存模型增加一个轻量标记，例如 `needsRefinalize`，以支持更准确的状态展示。

### 5. `src/components/panels/sidebar/ProjectTree.tsx`

需要确认一个产品选择：

- 当章节从 `finalized` 回退到 `revised` 后，是否应该立即从“正文章节”列表移除？

推荐：

- 是，移除。

原因：

- “正文章节”应只展示当前稳定终稿。
- 若章节已返工但仍保留在“正文章节”，会误导用户认为当前正文依然是稳定基线。

实现影响：

- 现有逻辑已按 `status === 'finalized'` 过滤，无需改动。
- 只需确认这是预期行为，并在交互文案中提前说明。

### 6. `electron/controllers/db-controller.ts`

本期最小实现：

- 继续复用 `db:draft-update-status`。

第二阶段建议：

- 新增专用 IPC：`db:draft-reopen-finalized`

职责：

- 在主进程侧做“仅最新定稿章节可回退”的业务校验。
- 成功后统一执行状态回退。

原因：

- 当前仅靠前端 guard 无法形成真正的数据层保护。
- 若未来存在其他入口，仍可能绕过 UI 直接修改终稿状态。

### 7. `electron/repositories/draft-repository.ts`

本期最小实现：

- 不改 Repository 逻辑，仍由现有 `updateStatus()` 承担状态更新。

第二阶段建议：

- 在 Repository 或更上层 Service 增加显式的“返工终稿”业务方法。
- 让“普通改状态”和“终稿返工”在数据层成为两类不同动作。

## 数据一致性

### 定稿后处理状态

返工一旦发生，旧的定稿后处理结果在业务上就已经过期。

本期建议：

- 当章节退出 `finalized` 后，不再展示“定稿后处理状态面板”。
- 因为当前面板已经只在 `status === 'finalized'` 时展示，这一点天然成立。

需要接受的现实：

- 底层状态文件或旧的后处理结果数据可能仍然保留。
- 但只要 UI 在返工后不再把这些结果呈现为“当前有效”，用户认知上就是安全的。

### 根目录物理文件

返工后，旧的根目录章节 txt 会暂时保留，直到重新定稿时被覆盖。

这意味着：

- 文件系统中会短暂存在“上一版终稿文件”。
- 这是可接受的，因为它代表最近一次稳定定稿产物，而不是当前编辑中的草稿。

## 风险与取舍

### 已接受风险

- 复用 `revised` 会让“普通修稿”和“终稿返工”在底层状态值上共用同一编码。
- 在没有新增标记字段的前提下，侧栏状态文案可能暂时无法精准区分这两种情况。

### 不接受风险

- 不接受历史定稿章节可返工。
- 不接受 `finalized` 直接长期可写。
- 不接受返工后自动重跑定稿后处理。

## 测试策略

建议覆盖以下场景：

1. 最新定稿章节点击“重新编辑”成功，状态退回 `revised`，编辑器恢复可写。
2. 非最新定稿章节点击“重新编辑”被拦截，提示正确。
3. 返工后章节从“正文章节”中消失。
4. 返工后继续写下一章时，被“前一章未定稿”规则正确拦截。
5. 返工后再次点击“定稿/重新定稿”，原有定稿流程能正常完成。
6. 重新定稿完成后，章节重新出现在“正文章节”列表中。

## 实施顺序

1. 在 `workflow-guards.ts` 新增 `guardReopenFinalizedChapter()`。
2. 在 `DraftEditor.tsx` 增加“重新编辑”按钮与 `doReopenFinalize()`。
3. 保持章节退回到 `revised` 后恢复可编辑。
4. 调整确认文案和“重新定稿”文案。
5. 验证返工后“正文章节”投影和写稿 guard 的联动是否符合预期。
6. 如体验需要，再评估是否增加 `needsRefinalize` 一类轻量标记做第二阶段优化。

## 推荐结论

采用“仅最新定稿章节允许返工，返工后退回 `revised`，改完后手动重新定稿”的最小可行方案。

该方案的优点是：

- 与现有状态链和 guard 语义最一致。
- 改动范围集中在 UI 和 guard，风险可控。
- 不需要重写定稿工作流。
- 能在不破坏系统基本假设的前提下，满足“终稿可返工”的真实需求。
